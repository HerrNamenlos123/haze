# Reactive Arrays — Implementation Guide

This document captures a full design session. A future Claude instance reading this should be able to implement reactive dynamic arrays without re-exploring the codebase. Everything in this file was derived by reading the actual source.

## 3. The Dynamic Array Type System

### Semantic IR (`SemanticTypes.ts`)

```typescript

  datatype: TypeUseId;       // element type

```

Created by `makeDynamicArrayDatatypeAvailable()` in `LookupDatatype.ts`. The `.length` synthetic field is registered there.

### C representation

`hzstd_dynamic_array_t*` — a pointer to the array control struct:
```c
typedef struct {
  void* buffer;
  size_t elem_size;
  size_t size;       // current number of elements (== .length in Haze)
  size_t capacity;
} hzstd_dynamic_array_t;
```

C typedef generated by CodeGenerator: `typedef hzstd_dynamic_array_t* __hz_DARRAY_<mangled_element>;`

### Array member access (Elaborate.ts lines 7046–7305)

In `resolveMemberAccess()`, after the reactive unwrap at line 6888, when the type is `DynamicArrayDatatype`:

- **`.length`** → `memberAccessRaw(exprId, "length", intType, true, sourceloc)`
- **`.push(element)`** → creates synthetic function `__hz_dynamic_array_push_<T>` with body `__c__("HZSTD_ARRAY_PUSH(this, <T>, element);");`
- **`.pop()`** → creates synthetic function `__hz_dynamic_array_pop_<T>` with body `__c__("return HZSTD_ARRAY_POP(this, <T>);");`
- **`.reserve(capacity)`** → synthetic function, body `__c__("hzstd_dynamic_array_reserve(this, capacity);");`

All synthetic functions use `syntheticFunctionFromCode()` from SemanticBuilder.ts. They are cached in `this.sr.syntheticFunctions` keyed by function name.

### Array subscript (Elaborate.ts `arraySubscript()`, line 10389)

Handles both slices and single-index access. For `DynamicArrayDatatype`, generates `ArraySubscriptExpr`. Does NOT unwrap reactive before this — the reactive unwrap must be handled here too (see §6 for why this matters).

### Array creation (Lower.ts lines ~1342–1430)

Dynamic array literals: heap-allocated via `HZSTD_DYNAMIC_ARRAY_CREATE`, then elements pushed one by one.

## 4. The Reactive Type System

### Semantic IR

```typescript
// ENode.ReactiveDatatype
export type ReactiveDatatypeDef = {
  variant: ENode.ReactiveDatatype;
  wrappedType: TypeUseId;   // what's inside the reactive wrapper
  concrete: boolean;
};

// ENode.ShallowReactiveDatatype — same shape, different semantics
// ENode.ComputedDatatype — same shape
```

### C representation

- `Reactive<T>` → `typedef hzstd_reactive_cell_t* __hz_reactive_<mangled_T>;`
- `ShallowReactive<T>` → same
- `Computed<T>` → `typedef hzstd_computed_node_t* __hz_computed_<mangled_T>;`

The reactive cell:
```c
struct hzstd_reactive_cell_t {
  hzstd_node_t base;   // dependents list (graph edges)
  void* value;         // pointer to heap-allocated slot holding the actual value
};
```

Key C API:
- `hzstd_reactive_create(void* initial)` — create cell; initial is a heap slot pointer
- `hzstd_reactive_read(cell)` — read slot pointer AND register current computed as dependent
- `hzstd_reactive_write(cell, void* slot)` — update slot AND notify all dependents (marks them dirty)
- `HZSTD_REACTIVE_CREATE(reactive_T, value_T, value)` — allocates slot, copies value, wraps in cell
- `HZSTD_REACTIVE_READ(value_T, cell)` — reads and copies value from slot (also registers dependency)
- `HZSTD_REACTIVE_WRITE(reactive_T, value_T, cell, value)` — writes new value into slot and notifies

### How computed/tracking works

A global `g_current_computed` pointer tracks the currently-evaluating computed node. When `hzstd_reactive_read(cell)` is called while a computed is evaluating, the cell registers the computed as a dependent. When `hzstd_reactive_write(cell, ...)` is called, all registered dependents are marked dirty. The next time the computed is accessed, it re-evaluates.

### Reactive unwrapping in Elaborate.ts

`unwrapReactiveOrComputedIfPossible(exprId)` (line 165) — inserts a `ReactiveReadExpr` node via `Conversion.MakeConversion` with plan `"reactive-read"`. The resulting expression has the WRAPPED type. In Lower.ts, `ReactiveReadExpr` lowers to `makeIntrinsicCall("HZSTD_REACTIVE_READ", [typeNameExpr, valueExpr], valueTypeId)`.

**Critical**: line 6888 in `resolveMemberAccess()`:
```typescript
exprId = this.unwrapReactiveOrComputedIfPossible(exprId);
```
This fires for ALL dot-access on any reactive value. For reactive arrays, we must **intercept before this line** to generate reactive-array-aware code instead.

### Reactive write in Elaborate.ts

Assignment `lhs = rhs` where `lhs` is of type `Reactive<T>` generates `ReactiveWriteExpr` (line 5745). The value is converted to the wrapped type `lhsType.wrappedType`. This is how `:=` on reactive fields works.

## 6. The Problem Being Solved

When user writes:
```haze
let arr = rx.reactive<[]int>([1, 2, 3]);
arr.push(4);  // should invalidate computed that read arr.length or arr[i]
```

**Current broken behavior:**
1. `arr: Reactive<[]int>` — wrapped in a reactive cell storing a `hzstd_dynamic_array_t*` pointer
2. `arr.push(4)` → `unwrapReactiveOrComputedIfPossible` → `HZSTD_REACTIVE_READ` gives back the array pointer → calls plain `HZSTD_ARRAY_PUSH(array_ptr, int, 4)` which mutates the array in place
3. **No reactive notification ever fires** because `hzstd_reactive_write` is never called
4. `arr.length` and `arr[i]` also unwrap first, so **no dependency is registered** on the reactive cell — subscript tracking is broken

---

## 7. The Solution Design

### Core idea

`Reactive<[]int>` is silently **promoted** by the compiler to `Reactive<[](Reactive<int>)>`. The user never writes this; the compiler applies it at type elaboration time, exactly as struct fields become reactive in `Reactive<Struct>`.

Two independent reactive layers:
1. **Version counter** (one `hzstd_reactive_cell_t*` per reactive array) — tracks structural changes: push, pop, clear, reading length. Subscript reads also track on this.
2. **Per-element cells** — each element stored as `Reactive<T>` (i.e., `hzstd_reactive_cell_t*`). Element-level reactivity is independent.

`ShallowReactive<[]int>` is NOT promoted — elements stay `int`. Only the version counter exists.

### New C struct

```c
typedef struct hzstd_reactive_array_t {
    hzstd_dynamic_array_t* data;    // stores elements of type Reactive<T> = hzstd_reactive_cell_t*
    hzstd_reactive_cell_t* version; // stores size_t counter; bump on structural mutation
} hzstd_reactive_array_t;
```

The version cell stores a `size_t` counter. To "bump": read the current value from the slot, increment, write back via `hzstd_reactive_write` to notify all dependents.

### C macros needed

```c
// Track: register dependency on version (for reads like subscript, length)
HZSTD_REACTIVE_ARRAY_READ_TRACK(ra)
  → hzstd_reactive_read(ra->version)   // discard return; side-effect is registration

// Mutations (bump = increment counter, then reactive_write to notify)
HZSTD_REACTIVE_ARRAY_PUSH(ra, elemType, elem)     → push to ra->data + bump version
HZSTD_REACTIVE_ARRAY_POP(ra, elemType)            → pop from ra->data + bump version, return elem
HZSTD_REACTIVE_ARRAY_CLEAR(ra)                    → clear ra->data + bump version
HZSTD_REACTIVE_ARRAY_SET(ra, index, elemType, val)→ set element + bump version

// Reads (register dep + access)
HZSTD_REACTIVE_ARRAY_GET(ra, index, length)       → track + subscript ra->data
HZSTD_REACTIVE_ARRAY_LENGTH(ra)                   → track + return ra->data->size

// Creation
hzstd_reactive_array_t* hzstd_reactive_array_create(hzstd_dynamic_array_t* data)
  → alloc struct, create version cell with zero counter, set data
```

Bump helper (can be inline or macro):
```c
static inline void hzstd_reactive_array_bump(hzstd_reactive_array_t* ra) {
    size_t* slot = (size_t*)hzstd_reactive_read(ra->version);  // just get slot ptr (not tracking; but ok)
    (*slot)++;
    hzstd_reactive_write(ra->version, slot);
}
```
Note: calling `hzstd_reactive_read` here for `bump` is fine — it will also register the computed if one is active, but that's harmless (it'll just be bumped again). Alternatively, access `ra->version->value` directly to avoid the tracking side-effect in bump.

### Step 3: Type promotion for `Reactive<DynamicArray<T>>`

When the compiler elaborates `Reactive<T>` where T is a `DynamicArrayDatatype<U>`, the stored `wrappedType` should be `DynamicArrayDatatype<Reactive<U>>` rather than `DynamicArrayDatatype<U>`.

**Where to do this:** In `LookupDatatype.ts` in `makeReactiveDatatypeAvailable()` (or wherever `Reactive<T>` types are constructed). When the wrapped type is `DynamicArrayDatatype`, create/look up `DynamicArrayDatatype<Reactive<elementType>>` and use that as the actual wrapped type.

For `ShallowReactive<DynamicArray<T>>`: do NOT promote — elements stay `T`.

This means:
- `Reactive<[]int>` has `wrappedType` = `[]Reactive<int>` internally
- Subscripting the inner array naturally yields `Reactive<int>` — no special case needed at subscript level for element type
- `.push()` signature on the inner array type expects `Reactive<int>` — naturally correct

**Important**: The USER still writes `Reactive<[]int>`, and the HAZE TYPE NAME still shows as `Reactive<[]int>` in error messages. Only the internal `wrappedType` pointer differs. The promotion is invisible to the user.

### Step 4: `stdlib/core/src/hzstd/hzstd_array.h` — Add reactive array C struct and macros

Add (at end of file, after including hzstd_reactive.h or in a new `hzstd_reactive_array.h` that hzstd.h includes):

```c
// === Reactive Array ===

typedef struct hzstd_reactive_array_t {
    hzstd_dynamic_array_t* data;
    hzstd_reactive_cell_t* version;
} hzstd_reactive_array_t;

static inline hzstd_reactive_array_t* hzstd_reactive_array_create(hzstd_dynamic_array_t* data) {
    hzstd_reactive_array_t* ra = (hzstd_reactive_array_t*)hzstd_heap_allocate(sizeof(hzstd_reactive_array_t));
    ra->data = data;
    size_t zero = 0;
    void* slot = hzstd_slot_alloc(sizeof(size_t));
    hzstd_slot_write(slot, &zero, sizeof(size_t));
    ra->version = hzstd_reactive_create(slot);
    return ra;
}

static inline void hzstd_reactive_array_bump(hzstd_reactive_array_t* ra) {
    // Access slot directly to avoid triggering dependency registration
    size_t* counter = (size_t*)(ra->version->value);
    (*counter)++;
    hzstd_reactive_write(ra->version, ra->version->value);
}

#define HZSTD_REACTIVE_ARRAY_READ_TRACK(ra) \
    ((void)hzstd_reactive_read((ra)->version))

#define HZSTD_REACTIVE_ARRAY_LENGTH(ra) \
    ({ HZSTD_REACTIVE_ARRAY_READ_TRACK(ra); (hzstd_int_t)(ra)->data->size; })

#define HZSTD_REACTIVE_ARRAY_PUSH(ra, elemType, elem) \
    (void)({ \
        HZSTD_ARRAY_PUSH((ra)->data, elemType, elem); \
        hzstd_reactive_array_bump(ra); \
    })

#define HZSTD_REACTIVE_ARRAY_POP(ra, elemType) \
    ({ \
        elemType __hz_popped = HZSTD_ARRAY_POP((ra)->data, elemType); \
        hzstd_reactive_array_bump(ra); \
        __hz_popped; \
    })

#define HZSTD_REACTIVE_ARRAY_CLEAR(ra) \
    (void)({ \
        hzstd_dynamic_array_clear((ra)->data); \
        hzstd_reactive_array_bump(ra); \
    })

#define HZSTD_REACTIVE_ARRAY_GET(ra, index, length) \
    *({ \
        HZSTD_REACTIVE_ARRAY_READ_TRACK(ra); \
        hzstd_int_t __hz_idx = (index); \
        if (__hz_idx < 0 || __hz_idx >= (hzstd_int_t)(length)) { \
            HZSTD_PANIC_FMT("array index out of range [%" PRId64 "] with length %" PRId64, \
                            __hz_idx, (hzstd_int_t)(length)); \
        } \
        &((elemType*)(ra)->data->buffer)[__hz_idx]; \
    })

// HZSTD_REACTIVE_ARRAY_GET doesn't know element type — use HZSTD_DYNAMIC_ARRAY_GET on ra->data after tracking
// Alternative: HZSTD_REACTIVE_ARRAY_GET_TYPED
#define HZSTD_REACTIVE_ARRAY_GET_TYPED(ra, elemType, index) \
    ({ \
        HZSTD_REACTIVE_ARRAY_READ_TRACK(ra); \
        HZSTD_DYNAMIC_ARRAY_GET((ra)->data, elemType, index); \
    })

#define HZSTD_REACTIVE_ARRAY_SET(ra, index, elemType, val) \
    (void)({ \
        elemType __hz_val = (val); \
        hzstd_dynamic_array_set((ra)->data, (size_t)(index), &__hz_val); \
        hzstd_reactive_array_bump(ra); \
    })
```

Note: `hzstd_heap_allocate` is available from hzstd_memory.h. The `hzstd_reactive_array.h` should include `hzstd_reactive.h` and `hzstd_array.h`.

Make sure `hzstd.h` includes the new header.

This method creates synthetic functions that take `hzstd_reactive_array_t*` as `this` and call the `HZSTD_REACTIVE_ARRAY_*` macros:

- **`pop()`**: Synthetic function returning `Reactive<T>`, body `__c__("return HZSTD_REACTIVE_ARRAY_POP(this, <elemType>);");`

- **`clear()`**: body `__c__("HZSTD_REACTIVE_ARRAY_CLEAR(this);");`

The `this` type for these synthetic functions is the REACTIVE ARRAY TYPE (`exprTypeUse.type`), not the unwrapped inner array type. So `envType.thisExprType` is the reactive array type.

Cache keys: `__hz_reactive_array_push_<elementTypeMangle>`, etc.

## 9. Type Promotion Deep Dive

The type promotion `Reactive<[]T>` → `Reactive<[]Reactive<T>>` means:

When `makeReactiveDatatypeAvailable(sr, wrappedTypeId, ...)` is called for a `DynamicArrayDatatype<T>`:
1. Look up `Reactive<T>` type (recursively call `makeReactiveDatatypeAvailable` for T)
2. Look up `DynamicArrayDatatype<Reactive<T>>` (call `makeDynamicArrayDatatypeAvailable` with the reactive element type)
3. Use this promoted array type as the actual `wrappedType` for the new `ReactiveDatatypeDef`

After this, in the type system, `Reactive<[]int>.wrappedType` = `[]Reactive<int>`, so:
- The C typedef for `Reactive<[]int>` checks: wrappedType is `DynamicArrayDatatype` → emits `hzstd_reactive_array_t*` ✓
- The member access after unwrapping would see `[]Reactive<int>` — but we intercept BEFORE the unwrap anyway in Step 7, so this is mostly relevant for correctness of push/pop signatures
- `.push(x)` where the array is `[]Reactive<int>` expects a `Reactive<int>` argument ✓

For `ShallowReactive<[]int>`: NO promotion. `wrappedType` stays `[]int`. But C typedef still needs to emit `hzstd_reactive_array_t*` (because the version counter is needed). The check in Step 6 needs to also catch `ShallowReactive<DynamicArray<T>>`.

For `ShallowReactive`, subscript returns plain `T` (no element cells). Push takes plain `T`. The only reactive layer is the version counter.

## 11. Synthetic Function Pattern (for reference)

All array methods are created via this pattern in Elaborate.ts:

```typescript
const funcname = `__hz_reactive_array_push_${Semantic.mangleTypeUse(this.sr, elemTypeId).name}`;

if (!this.sr.syntheticFunctions.has(funcname)) {
  const functionType = makeRawFunctionDatatypeAvailable(this.sr, {
    parameters: [{ optional: false, type: elemTypeId }],
    returnType: this.sr.b.voidType(),
    requires: { final: true, pure: false, noreturn: false, noreturnIf: null },
    sourceloc,
    vararg: false,
  });

  const elemName = Semantic.mangleTypeUse(this.sr, elemTypeId);
  const code = `__c__("HZSTD_REACTIVE_ARRAY_PUSH(this, ${
    elemName.wasMangled ? "_H" + elemName.name : elemName.name
  }, element);");`;

  const [func, funcId] = this.sr.b.syntheticFunctionFromCode({
    functionTypeId: functionType,
    parameterNames: ["element"],
    funcname,
    bodySourceCode: code,
    currentScope: this.currentContext.currentScope,
    sourceloc,
    envType: {
      type: "method",
      thisExprType: makeTypeUse(
        this.sr,
        exprTypeUse.type,          // the Reactive<[]int> type (not the inner array)
        EDatatypeMutability.Mut,
        "force-no-inline",
        sourceloc
      )[1],
    },
  });
  this.sr.syntheticFunctions.set(funcname, funcId);
}

// Return callableExpr wrapping the function
return this.sr.b.callableExpr(funcId, { type: "method", thisExprType: ... }, { type: "method", thisExpr: exprId }, sourceloc);
```

The `thisExprType` in `envType` and `callableExpr` must match — it's the type of `this` (the reactive array expression, not the unwrapped array).

## 13. Things to Watch Out For

1. **`makeRawFunctionDatatypeAvailable` import**: Already imported in Elaborate.ts; check the import if you add a new method.

2. **`exprTypeUse` vs `resolvedExprTypeUse`**: In Elaborate.ts, after line 6888, `exprTypeUse` is the type of the UNWRAPPED expression. For the reactive array intercept (before line 6888), use the original type before unwrapping.

3. **The `meta.TypeCategory` C enum**: The Haze meta.hz enum is backed by a C enum `hzstd_meta_type_category_t`. Search for this in the hzstd headers — it's likely in `hzstd_common.h` or similar. The integer values must match between the C enum and the Haze enum.

4. **Export.ts**: If any new data is added to symbol nodes or type def nodes that needs to survive cross-module compilation (e.g., a flag on `ReactiveDatatypeDef` indicating it wraps a dynamic array), it must be serialized in `src/SymbolCollection/Export.ts`. For this feature, the type promotion is re-derived at elaboration time, so no new Export.ts changes should be needed — the promoted type is reconstructed from the structure.

5. **`T.elementType` in reactive.hz**: The comptime meta accessor `T.elementType` for array types may or may not exist. Search Elaborate.ts for `elementType` in the T-meta resolution section. If it doesn't exist, add it alongside `T.length`, `T.fields`, etc.

6. **`hzstd_reactive_read` side effects in bump**: The `hzstd_reactive_read` function REGISTERS a dependency when called during computed evaluation. In `hzstd_reactive_array_bump`, we should NOT register a dependency (it's a write path). Access `ra->version->value` directly or use a lower-level helper that doesn't register. See `hzstd_reactive.c` for `hzstd_reactive_read` — it calls `register_dependency`.

7. **Reactive array as `this` in synthetic functions**: The C `this` variable in synthetic function bodies has the Haze type of the reactive array (e.g., `hzstd_reactive_array_t*`). So `HZSTD_REACTIVE_ARRAY_PUSH(this, elemType, element)` works correctly.

8. **ShallowReactive array typedef**: The check in CodeGenerator.ts must also handle `ShallowReactiveDatatype`. Currently `ShallowReactiveDatatype` may share the `ReactiveDatatype` typedef branch or have its own. Check lines around 641 for this.

## 14. What Does NOT Need to Change

- `Lower.ts` — `ReactiveDatatype` lowering (line 2868) is fine as-is; it lowers the type by calling `lowerTypeUse` on the wrapped type, which will correctly lower the promoted `[]Reactive<int>` type
- The existing `ReactiveReadExpr` lowering — not used for reactive arrays (we intercept before the unwrap)
